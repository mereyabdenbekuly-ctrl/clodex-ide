# Clodex — Product Overview

## What It Is

Clodex is a local agentic IDE for software development.

It brings together:

- an AI-agent chat;
- file and Git operations;
- a terminal;
- an integrated browser;
- models from multiple providers;
- MCP, skills, and plugins;
- local, Docker, SSH, and cloud execution;
- long-term task memory;
- secure applications and tools generated by the agent.

The user describes the desired outcome, and the agent analyzes the project,
plans the work, modifies code, runs checks, and presents the completed result.
Risky actions remain under the control of the user and the security policies.

## Typical Workflow

1. The user opens a project.
2. They create a task and describe the expected result.
3. The agent analyzes the project structure and instructions.
4. It selects an appropriate model.
5. It reads files, runs commands, and modifies code.
6. It records important facts and decisions.
7. It requests confirmation before risky actions.
8. It runs tests and other checks.
9. It presents the diff and final result.
10. The task can continue after a restart or move to another environment.

## Core Capabilities

### AI Tasks

- create, search, continue, and archive tasks;
- goals, progress tracking, and budgets;
- fork tasks while preserving context;
- concise status updates during execution;
- a separate, self-contained final result.

### Code

- read and modify files;
- search across the project;
- Git and worktrees;
- diff review;
- accept or reject changes;
- review GitHub pull requests;
- protected merge operations.

### Execution

- local terminal;
- Docker runner;
- SSH runner;
- custom runners;
- cloud tasks;
- artifacts and execution receipts.

### Browser

- tabs and navigation;
- visual context;
- permissions;
- network destination controls;
- audit log for allowed and blocked requests.

### Models

- Clodex Gateway;
- bring-your-own API keys;
- OpenAI-compatible endpoints;
- local models;
- task-aware model selection;
- token, budget, and quota tracking.

### Memory

- complete task history;
- important facts and user constraints;
- links between facts and their sources;
- stale-information detection;
- contradiction handling;
- short-term and long-term summaries;
- recovery after restart.

### Extensions

- MCP;
- skills;
- plugins;
- private plugin sources;
- scheduled automations;
- applications generated by the agent.

## Security

Clodex does not rely on model instructions alone:

- a model cannot grant permissions to itself;
- commands and network access are evaluated by a separate policy layer;
- authorization is bound to a specific action;
- corrupted or invalid context is blocked;
- generated applications run in a sandbox;
- secrets are not exposed directly to extensions;
- telemetry must not contain prompts, files, commands, or credentials;
- remote execution is bound to snapshots, leases, and receipts.

## What Makes the Product Different

1. It is a complete IDE, not a standalone chat window.
2. Long-running task context is preserved.
3. Local, Docker, SSH, and cloud execution are supported.
4. Users can choose providers and models.
5. Model decisions are separated from authorization decisions.
6. The platform is extensible through MCP, plugins, skills, and runners.
7. New capabilities are enabled only after validation and real-world evidence.

## Current Status

The architectural core is implemented:

- tasks and Agent Core;
- memory;
- model routing;
- security policy;
- network controls;
- local and remote execution;
- session transfer;
- generated-application runtime;
- release-readiness infrastructure.

Advanced capabilities remain disabled until sufficient real-world observations
and approval are available.

Before production release, the project still requires:

- an official protected repository;
- release signing;
- observations from real installations;
- macOS, Windows, and Linux acceptance testing;
- a canary rollout;
- monitoring and rollback procedures.

## What to Test First

1. A standard coding task.
2. File operations and diff review.
3. The local terminal.
4. Restart and task recovery.
5. Connecting a model provider.
6. Network Egress settings.
7. An MCP server.
8. An SSH or Docker runner.
9. Generated Apps.
10. A large, multi-stage task that relies on memory.
