# Clodex `v1.16.0-preview.2` rollback-baseline plan

This committed schema-v2 plan prepares a signed and notarized rollback
baseline. It does not dispatch a workflow, create a tag, publish a public
download, or claim acceptance evidence.

## Immutable role

- source branch: `main` (resolved to an immutable SHA at dispatch);
- version/tag: `1.16.0-preview.2` / `v1.16.0-preview.2`;
- promotion role: `rollback-baseline`;
- GitHub Release state: **draft** behind the protected `Release` environment;
- public links: disabled;
- canary installations: `0` (operator acceptance is not a canary);
- rollback mode: `distribution-stop-only`;
- rollback target tag: **none**;
- manifest: `.release-notes/clodex-technical-preview.json`.

`v1.16.0-preview.1` is preserved as historical state, but it is not trusted
rollback evidence: its tag, release target, artifact provenance, and displayed
application version are not consistently bound. Never move, delete, rewrite,
or advertise it as the rollback target.

Public website download surfaces must remain disabled for preview.1 and its
static DMG alias. Preview.2 is a protected draft and is not a replacement
public download.

The website temporarily redirects the historical DMG path to release status.
Deployment owners must also purge or deny the legacy object at the external
Cloudflare/origin layer; the application redirect is not proof that the binary
has been removed from every upstream cache.

## Dispatch sequence

Only after external signing readiness is genuinely green:

```bash
gh workflow run technical-preview-release.yml \
  --repo mereyabdenbekuly-ctrl/clodex-ide \
  --ref main \
  -f ref=main \
  -f manifest=.release-notes/clodex-technical-preview.json \
  -f confirm=RELEASE
```

The workflow validates the committed manifest, pins the candidate SHA, runs
the protected promotion gates, creates an immutable tag, builds/signs/notarizes
all platform artifacts, validates manifest-bound assets, and stages a **draft**
GitHub Release. It does not make preview.2 public.

Every platform emits a schema-v2 validation manifest plus SHA-256 evidence:

- `macos-{arm64,x64}-1.16.0-preview.2.{json,sha256}`;
- `windows-x64-1.16.0-preview.2.{json,sha256}`;
- `linux-x64-1.16.0-preview.2.{json,sha256}`.

## Acceptance and committed evidence

The protected publication workflow binds the four validation manifests and the
exact live asset set to the committed manifest/source, then attests the
aggregate report bytes. The protected acceptance workflow verifies that
attestation, re-runs real source checks, and attests the schema-v4 evidence.
Hand-written committed JSON is never evidence.

Preview.2 can reach only `ready-as-rollback-baseline`. It cannot become
`ready-for-canary`, cannot contain canary telemetry, and cannot name preview.1
as a target. Only exact evidence bytes that pass `gh attestation verify` for
the protected acceptance workflow may be committed as
`.release-evidence/v1.16.0-preview.2.json`.

## Next chain links

1. Create the preview.3 canary manifest only after the real, attested preview.2
   report is committed.
2. The validator recomputes preview.2's historical manifest hash from its
   source commit and requires the prerequisite tag to resolve to that commit.
3. Preview.3 is also a draft release. Its controlled distribution is limited
   to exactly five installations for at least 24 hours.
4. Preview.3 and stable promotion remain `NOT_READY` until a separately
   attested, manifest-bound distribution/telemetry observation source replaces
   the current hard blocker; operator JSON is never sufficient.
5. Create `.release-notes/clodex-stable.json` only after the real preview.3
   report proves a trusted closed >=24-hour exactly-five window, reaches
   `ready-for-stable`, passes attestation verification, and is committed.
6. Stable `clodex@1.16.0` is built anew; preview artifacts are never relabeled.

Promotion validators live-revalidate the draft release, assets, exact remote
tag, and complete preview.3 to preview.2 chain. They also reject product-code
changes after an accepted source commit. Only release manifests/evidence, the
changelog, and release documentation may change between links; any code delta
restarts the chain.

Stopping distribution keeps the affected draft release non-public and halts
any controlled out-of-band delivery. Electron updates remain forward-only for
clients that already installed a candidate.
